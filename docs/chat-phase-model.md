# 实时聊天 - 分阶段模型（黄总 20:01 最终确认）

## 阶段状态

### 阶段一：下单后 → 预约时间前
- **用户侧**：订单确认页，显示倒计时，**无聊天框**
- **师傅侧**：后台显示"补充留言区"，用户可提交500字背景信息
- **交互模式**：单向提交（用户→师傅），师傅可提前阅读准备

### 阶段二：预约时间到
- **用户侧**：聊天窗口**弹出开放**，双方实时对话
- **师傅侧**：进入实时咨询状态
- **交互模式**：双向实时（文字/视频/语音）

### 阶段三：咨询结束
- **用户侧**：窗口关闭，历史消息归档到订单详情
- **师傅侧**：订单标记完成，收入结算
- **交互模式**：只读历史

## 技术选型（针对分阶段模型调整）

### 主推荐：Daily.co（视频+聊天）

**为什么更适合分阶段模型：**
1. **会话有明确生命周期**：Daily房间 = 预约时间窗口，到点开房、结束关房
2. **无需持久连接**：不像客服需要24小时在线，Daily房间只在预约时段存在
3. **命理场景必须视频**：八字排盘展示、塔罗抽牌直播、风水远程看房
4. **师傅提前准备**：Daily Prebuilt支持"等待室"模式，师傅可提前进入、用户到点加入
5. **2000分钟免费** = 约60次30分钟咨询，够用3-6个月

**实现路径：**
```
下单 → 用户写补充留言（500字）→ 师傅后台查看
        ↓
预约时间到 → Daily API创建房间 → 用户弹窗加入
        ↓
咨询中 → 视频+文字实时 → 师傅标记完成
        ↓
结束后 → Daily房间关闭 → 聊天记录归档
```

**环境变量：**
```
DAILY_API_KEY=your_daily_api_key
DAILY_DOMAIN=your_domain.daily.co
```

### 备选：WebSocket自建（纯文字）

**适用场景**：如果黄总认为纯文字足够

**实现路径：**
1. Next.js API Routes + Socket.io 或 ws
2. Supabase Realtime 作为备选（更简单）
3. 预约时间到，前端建立WebSocket连接
4. 消息存储到 Supabase messages 表

**缺点**：
- 需要自建UI（发送/接收/在线状态/历史记录）
- 无视频能力（命理场景受限）
- 开发量 > Daily.co

## UI原型清单

1. **预约前页面** (`/booking/confirmation`)：倒计时 + 补充留言表单
2. **补充留言区** (`/master/orders/[id]`)：师傅后台查看用户留言
3. **聊天弹窗** (`/components/chat/consultation-chat.tsx`)：预约到点弹出
4. **订单归档** (`/user/orders/[id]`)：咨询结束后查看历史

## 决策建议

**选 Daily.co**，理由：
- 分阶段模型天然匹配 Daily 的"房间生命周期"概念
- 视频是命理咨询的核心体验，不是锦上添花
- 开发量最小（嵌入iframe或Prebuilt UI）
- 已有依赖 `@daily-co/daily-js`
